# 项目开发 Agent 规范

适用范围：本仓库根目录及所有子目录。

本规范用于约束 AI Coding Agent 与人工开发者在本项目中的开发行为。核心目标是在保证可维护性、可测试性与可扩展性的前提下，高质量完成项目开发，并持续优化已有代码结构，而不是简单堆功能。

## 1. 核心原则

一句话版本：

> 可测试、可复用、可拆分、禁止重复实现、禁止单体文件。

### 1.1 可测试优先

所有新增功能必须包含可验证的测试方式，优先级如下：

1. 单元测试
2. 最小可运行验证脚本
3. API 请求测试示例
4. 明确的输入/输出示例

要求：

- 每个核心逻辑必须可以独立验证。
- 测试必须独立于业务运行。
- 测试必须可重复执行。
- 禁止提交只能运行、无法验证的代码。

判断标准：

> 如果一个功能无法被单独验证，它就是不合格设计。

### 1.2 禁止重复实现

开发新功能前必须检查：

- 是否已有相似逻辑。
- 是否已有工具函数、service、组件或模块可复用。
- 是否可以通过扩展已有实现完成需求。

规则：

- 已实现功能必须复用，不得重复实现。
- 允许扩展，禁止复制再写一份。
- 如果复用了已有代码，交付说明中必须指出复用点。

### 1.3 模块化设计

禁止将所有逻辑堆叠在单一文件中。

必须遵守：

- 一个文件只负责一个职责。
- UI、service、data access、utils 必须分层。
- 页面只承载展示、交互调度与生命周期逻辑。
- 业务规则应放入 service 或 domain 模块。
- 通用纯函数应放入 utils，并配套可验证测试或样例。

禁止行为：

- 所有逻辑写在 `index.js`、`app.js` 或单一页面文件中。
- controller/page、business、data access 混写。
- 组件内部复制多份相同业务规则。

### 1.4 代码复用优先

所有重复出现两次以上的逻辑必须抽象。

抽象优先级：

1. `utils/` 纯函数
2. `services/` 业务服务
3. 通用组件
4. 模块级公共能力

禁止复制粘贴式开发。新增抽象必须有明确收益：减少真实重复、降低复杂度，或匹配项目现有结构。

### 1.5 渐进式开发

每次开发必须遵循：

- 小步改动。
- 功能可运行。
- 不破坏旧功能。
- 默认保持向后兼容，除非用户明确要求重构或破坏性升级。
- 修改已有接口时必须说明影响范围，并更新对应测试或验证方式。

## 2. 推荐代码结构

本项目是微信小程序项目。推荐按以下结构演进：

```text
project/
├── app.js
├── app.json
├── app.wxss
├── pages/
│   └── resume/
│       ├── index.js
│       ├── index.json
│       ├── index.wxml
│       └── index.wxss
├── components/
│   └── common-component/
├── modules/
│   ├── resume/
│   ├── user/
│   └── auth/
├── services/
│   ├── resumeService.js
│   └── userService.js
├── utils/
│   ├── date.js
│   └── validator.js
├── config/
│   └── env.js
└── tests/
    ├── resume.test.js
    └── validator.test.js
```

目录职责：

- `pages/`：页面入口、页面状态、用户交互与生命周期。
- `components/`：可复用 UI 组件，不直接承载复杂业务规则。
- `modules/`：按业务领域组织的模块能力。
- `services/`：业务流程、接口编排、跨模块协调。
- `utils/`：无业务状态的通用工具函数。
- `config/`：环境配置与常量配置。
- `tests/`：单元测试、验证脚本或输入输出样例。

## 3. Agent 开发流程

### 3.1 写代码前

Agent 在写代码前必须完成：

1. 检查是否已有类似功能。
2. 判断是否可以复用或扩展。
3. 设计新增代码的模块归属。
4. 确定测试或验证方式。

### 3.2 开发过程中

Agent 必须：

- 保持模块边界清晰。
- 抽象重复逻辑。
- 保证核心功能可独立运行和验证。
- 避免跨模块直接耦合。
- 优先遵循项目已有命名、目录与实现风格。

### 3.3 修改已有代码时

Agent 必须：

- 不破坏已有接口，除非用户明确要求。
- 优先扩展，不重写。
- 说明影响范围。
- 更新对应测试或验证说明。
- 不回滚用户已有改动，除非用户明确要求。

## 4. 测试规范

每个新增模块至少满足以下之一：

- 单元测试。
- 可执行 demo script。
- API 请求测试示例。
- 输入输出样例说明。

测试要求：

- 测试必须独立于业务运行。
- 测试必须可重复执行。
- 测试必须覆盖核心分支与异常输入。
- 禁止手动运行一次就算测试。

## 5. 质量红线

以下行为一律视为低质量实现：

- 单文件巨石代码。
- 没有任何测试或验证方式。
- 重复实现已有逻辑。
- 业务、数据、UI 混写。
- copy-paste 同功能代码。
- 不可复用设计。
- 为了短期完成需求而制造长期结构债务。

## 6. Agent 输出要求

每次交付代码时必须包含：

- 修改或新增模块说明。
- 使用方式。
- 测试方法。
- 影响范围说明。
- 复用点说明。

如果未能运行测试，必须明确说明原因和残余风险。
