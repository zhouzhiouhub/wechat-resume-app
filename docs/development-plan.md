# 简历小程序开发计划

## 1. 项目目标

简历小程序的核心目标是：让面试官在 30 秒内看到候选人最突出的能力。

因此本项目不追求功能堆叠，而是围绕“快速建立可信度”设计首页信息层级、项目展示路径和联系转化路径。

### 1.1 30 秒阅读路径

| 时间段 | 面试官要看到什么 | 对应功能 |
| --- | --- | --- |
| 0-5 秒 | 你是谁、当前定位、是否在寻找机会 | 个人名片、职位标签、状态标识 |
| 5-15 秒 | 你的核心技术能力是否匹配岗位 | 技能图谱、核心技能标签、能力分组 |
| 15-25 秒 | 你做过什么有结果的项目 | 项目作品画廊、项目截图、成果指标 |
| 25-30 秒 | 如何继续了解或联系你 | 一键联系、复制邮箱、二维码 |

### 1.2 成功标准

- 首页首屏必须展示姓名、头像、岗位、状态、核心技能和重点项目入口。
- 重点项目必须包含截图或动图、技术栈、难点、解决方案和量化成果。
- 联系方式必须在 2 次点击内完成复制、识别或发起联系。
- 所有核心数据先通过本地模块统一管理，后续可平滑迁移到云开发。
- 新增核心逻辑必须具备单元测试、验证脚本或明确输入输出样例。

## 2. 当前仓库状态

当前仓库已经具备微信小程序项目配置和开发规范：

- `project.config.json`：微信小程序项目配置。
- `README.md`：项目说明入口。
- `AGENTS.md`：项目开发规范，明确要求可测试、可复用、可拆分。

当前尚未实现小程序页面、组件、服务和测试。因此第一阶段应先搭建清晰目录结构和数据模型，避免后续把所有逻辑堆在页面文件中。

## 3. 功能优先级

### 3.1 P0：最小可用版本

P0 目标是先完成可以投递和展示的简历小程序，覆盖“30 秒看懂能力”的主路径。

| 功能 | 说明 | 模块归属 | 验证方式 |
| --- | --- | --- | --- |
| 个人名片 | 展示姓名、头像、岗位、状态和一句话优势 | `components/profile-card/` | 组件渲染输入输出样例 |
| 核心技能概览 | 按语言、框架、工程化分组展示技能 | `components/skill-overview/`、`modules/resume/` | 技能数据归一化单元测试 |
| 项目作品画廊 | 卡片展示项目截图、标题、角色、成果摘要 | `components/project-card/`、`pages/home/` | 项目列表数据测试 |
| 项目详情页 | 展示截图轮播、技术栈、难点、方案、成果 | `pages/project-detail/`、`services/resumeService.js` | 项目 ID 查询测试 |
| 一键联系 | 支持复制邮箱、展示微信二维码、客服按钮预留 | `components/contact-panel/`、`services/contactService.js` | 联系方式校验测试 |

### 3.2 P1：增强体验版本

P1 目标是补齐完整简历体验，并通过交互能力体现前端工程能力。

| 功能 | 说明 | 模块归属 | 验证方式 |
| --- | --- | --- | --- |
| 时间轴履历 | 展示教育、工作、实习经历 | `components/timeline/`、`modules/resume/` | 时间排序和日期格式测试 |
| 主题切换 | 支持浅色、暗黑和主题色切换 | `services/themeService.js`、`utils/theme.js` | 主题变量映射测试 |
| 分享海报 | 基于简历核心信息生成分享图片 | `pages/poster/`、`services/posterService.js` | 海报数据模型测试 |
| 打印版样式 | 输出适合面试现场展示的简洁版本 | `pages/print/` 或条件样式 | 页面状态切换样例 |

### 3.3 P2：数据能力版本

P2 目标是展示产品和数据意识，但只在 P0、P1 稳定后开发。

| 功能 | 说明 | 模块归属 | 验证方式 |
| --- | --- | --- | --- |
| 访问数据看板 | 记录访问量、停留时长、项目点击 | `services/analyticsService.js`、`pages/admin-dashboard/` | 埋点事件构造测试 |
| 隐藏入口 | 连续点击头像 5 次进入数据页 | `utils/tapCounter.js` | 连击计数边界测试 |
| 留言反馈板 | 允许访客留下问题或评价 | `services/feedbackService.js`、`pages/feedback/` | 表单校验和云接口样例 |

### 3.4 P3：后端展示版本

P3 目标是展示全栈能力，建议在小程序主体体验成熟后再做。

| 功能 | 说明 | 模块归属 | 验证方式 |
| --- | --- | --- | --- |
| 云数据库存储 | 存储访问记录、反馈内容和作品浏览记录 | `cloudfunctions/`、`services/` | 云函数请求示例 |
| 作品浏览通知 | 浏览重点项目后触发订阅消息 | `services/notificationService.js` | 订阅消息参数样例 |
| 管理端保护 | 数据看板仅自己可见 | `services/authService.js` | 权限判断测试 |

### 4.1 分层原则

- 页面层只负责页面状态、生命周期和用户交互调度。
- 组件层只负责展示和轻量交互，不直接写复杂业务规则。
- `modules/resume/` 负责简历领域数据结构、映射和默认数据。
- `services/` 负责业务流程、接口编排和微信 API 封装。
- `utils/` 只放无业务状态的纯函数，并优先配测试。

## 6. 关键页面设计

### 6.1 首页

首页承担主要转化任务，信息顺序建议固定为：

1. 个人名片：头像、姓名、岗位、状态、一句话优势。
2. 能力快照：3-5 个核心标签和技能分组概览。
3. 重点项目：优先展示 2-3 个最能证明能力的项目。
4. 履历时间轴：教育和工作经历摘要。
5. 联系入口：复制邮箱、二维码、客服按钮预留。

首页不放过长说明文字。每个模块都要有清晰标题、关键结果和跳转入口。

### 6.2 项目详情页

项目详情页用于支撑技术面试追问，建议结构为：

1. 项目截图轮播。
2. 项目背景和个人角色。
3. 技术栈标签。
4. 难点与解决方案。
5. 量化结果。
6. 代码、预览或联系入口。

### 6.3 分享海报页

分享海报只保留传播所需信息：

- 姓名、岗位、状态。
- 3 个核心技能标签。
- 1-2 个代表项目。
- 微信二维码或小程序码。

海报生成逻辑应由 `posterService.js` 输出标准绘制数据，页面只负责调用 Canvas 绘制。

### 6.4 数据看板页

数据看板仅本人可见，入口使用连续点击头像触发。看板先展示最小指标：

- 总访问量。
- 项目详情点击量。
- 平均停留时长。
- 联系按钮点击次数。

埋点事件统一由 `analyticsService.js` 生成，避免页面里散落事件名。

## 7. 开发里程碑

### M1：项目骨架与数据模型

- 创建小程序基础入口文件。
- 创建 `modules/resume/` 默认简历数据和模型映射。
- 创建 `services/resumeService.js`，统一读取简历信息。
- 创建基础测试或验证脚本，验证简历数据结构完整。

交付标准：

- 首页可以读取统一数据源。
- 简历数据字段缺失时有明确校验错误。
- 页面没有直接硬编码复杂简历数据。

### M2：P0 展示闭环

- 实现个人名片、技能概览、项目卡片、联系面板。
- 实现首页和项目详情页。
- 项目卡片点击跳转项目详情页。
- 联系面板支持复制邮箱和展示二维码。

交付标准：

- 面试官打开首页 30 秒内可以理解候选人定位、能力和代表项目。
- 重点项目详情信息完整。
- P0 组件可复用，不在页面中复制展示逻辑。

### M3：P1 交互增强

- 实现时间轴履历。
- 实现主题切换。
- 实现分享海报数据模型和页面。
- 补充主题和海报相关测试。

交付标准：

- 主题切换不影响内容可读性。
- 海报信息来自统一简历数据源。
- 时间轴按日期稳定排序。

### M4：P2 数据能力

- 实现访问埋点事件模型。
- 实现隐藏入口和数据看板页面。
- 可选接入云函数记录访问数据。
- 实现留言反馈板。

交付标准：

- 埋点事件统一命名并可测试。
- 数据看板与访客页面隔离。
- 留言内容有校验和权限边界。

### M5：P3 后端能力与发布打磨

- 接入云数据库和云函数。
- 接入订阅消息通知。
- 优化图片体积和首屏加载。
- 完成发布前检查。

交付标准：

- 云函数有请求样例和异常处理。
- 订阅消息失败不影响主流程。
- 首屏展示稳定，图片资源可正常加载。

## 8. 测试计划

### 8.1 单元测试重点

| 模块 | 测试重点 |
| --- | --- |
| `resumeService.js` | 获取个人信息、项目列表、项目详情、异常 ID |
| `resumeMapper.js` | 简历数据默认值、字段兼容、缺失字段处理 |
| `validator.js` | 邮箱、URL、日期、技能等级范围 |
| `themeService.js` | 主题名称、CSS 变量映射、默认主题 |
| `posterService.js` | 海报绘制数据是否包含必要字段 |
| `tapCounter.js` | 连续点击次数、超时重置、触发边界 |

### 8.2 最小验证脚本

在没有完整测试框架前，可以先提供 `tests/validateResumeData.js`：

```text
输入：modules/resume/resumeData.js
输出：字段完整性检查结果、项目 ID 唯一性检查结果、技能等级范围检查结果
```

### 8.3 小程序手动验证清单

- 首页首屏信息是否完整。
- 项目卡片是否能跳转详情页。
- 项目详情页是否能处理不存在的项目 ID。
- 邮箱复制是否成功。
- 二维码图片是否支持长按识别或保存。
- 主题切换后文字对比度是否正常。
- 分享海报是否能保存到相册。
- 隐藏入口是否不会被普通访问误触。

## 9. 复用策略

- 复用 `AGENTS.md` 中推荐的 `pages/`、`components/`、`modules/`、`services/`、`utils/`、`tests/` 分层结构。
- 复用统一简历数据源，首页、项目详情和海报都从 `resumeService.js` 获取数据。
- 复用 `project-card` 展示项目摘要，避免首页和列表页重复写卡片结构。
- 复用 `validator.js` 校验邮箱、日期、技能等级和项目链接。
- 复用 `themeService.js` 管理主题变量，组件只消费主题类名或变量。
- 复用 `analyticsService.js` 生成埋点事件名和参数，避免页面散落字符串。

## 10. 风险与取舍

- 分享海报、访问统计和订阅消息都可能增加开发复杂度，必须在 P0 展示闭环稳定后再做。
- 云开发能力能加分，但如果权限和异常处理不完整，反而会影响可靠性。
- 技能进度条不应虚高，建议用“掌握场景”和“项目证据”辅助说明。
- 项目截图比长文字更有说服力，应优先准备真实截图或动图。
- 首页不宜放太多模块，超过 30 秒主路径的信息应放入详情页。

## 11. 发布前检查清单

- `app.json` 页面路径清晰，命名统一。
- 页面文件没有承载复杂业务规则。
- 组件没有复制多份相同逻辑。
- 核心数据通过 `modules/resume/` 和 `services/resumeService.js` 管理。
- P0 核心能力具备测试或验证脚本。
- 所有图片资源可加载，重点项目有截图。
- 联系方式可用，邮箱复制和二维码识别路径清晰。
- README 已链接开发计划和开发规范。
